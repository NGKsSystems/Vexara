# AGENT RULES — PHASE RUNNER + PROOF LOCK

## 0. PURPOSE
All phases must run the same way.
No custom proof sprawl.
No improvising.
No “helpful” extras.

---

## 1. CORE EXECUTION RULE
Every task must follow this exact lifecycle:

1. Verify correct project root
2. Verify required inputs exist
3. Execute only the requested phase/task
4. Generate only final validation evidence
5. Report only final result
6. Stop

Do not add side quests.
Do not continue into the next phase.
Do not generate exploratory artifacts unless explicitly ordered.

---

## 2. ROOT GUARD (MANDATORY)
Before any action:
- verify current working directory exactly matches the required project root

If root is wrong:
- STOP immediately
- do not patch
- do not build
- do not generate proofs

---

## 3. FAIL-CLOSED RULE
If anything required is missing, unclear, broken, or out of bounds:
- STOP
- mark task FAIL
- report the exact blocker

Never guess.
Never silently recover.
Never switch strategies unless explicitly told.

---

## 4. MINIMAL CHANGE RULE
Only change what is required for the current task.

Forbidden unless explicitly requested:
- refactors
- cleanup passes
- naming rewrites
- formatting-only edits
- unrelated warnings
- “while I’m here” fixes
- touching nearby systems that are not part of the task

---

## 5. PHASE SCOPE LOCK
Each run may do only one of the following:
- one phase
- one bug
- one proof validation
- one targeted patch

If multiple tasks are present:
- do only the named/current task
- ignore future tasks until instructed

---

## 6. STANDARD OUTPUT CONTRACT
Every run must end with exactly this structure:

PHASE: [phase name or task name]
RESULT: PASS or FAIL

TASK:
[one line summary]

FILES CHANGED:
- [file1]
- [file2]
- [or NONE]

VALIDATION:
- [marker]=YES/NO
- [marker]=YES/NO

PF: [absolute path to final proof folder or NONE]
ZIP: [absolute path to final proof zip or NONE]
GATE: PASS or FAIL

No extra proof summaries.
No duplicate result blocks.

---

## 7. PROOF SYSTEM LOCK (STRICT)
Proof generation must be identical for every phase.

Allowed final proof outputs:
1. one final proof folder
2. one final proof zip

Optional:
- one single text log inside the proof folder if needed

Not allowed:
- multiple proof folders
- multiple zips
- exploratory logs
- retry logs kept as finals
- scratch evidence
- duplicate copies
- debug dumps unless explicitly requested

If more than the allowed final proof outputs are created:
- task is FAIL

---

## 8. FINAL PROOF CONTENT RULE
Final proof must contain only evidence that directly proves the current task.

Allowed contents:
- final runner output
- final validation markers
- required screenshots/logs only if essential
- exact files needed to prove PASS/FAIL

Forbidden:
- unrelated logs
- old logs
- duplicate logs
- entire build trees
- “just in case” evidence
- historical artifacts from prior runs

---

## 9. INTERMEDIATE ARTIFACT RULE
Intermediate artifacts are temporary only.

During execution:
- temporary logs may exist only if required

Before final output:
- remove, ignore, or quarantine intermediate artifacts
- leave only final accepted proof set

If cleanup cannot be done safely:
- report it
- do not pretend the proof is clean

---

## 10. SINGLE FINAL PROOF RULE
For an accepted PASS:
- only one final proof folder may remain as the active result
- only one final zip may remain as the active result

For a FAIL:
- only one failure proof folder may remain if needed
- only one failure zip may remain if needed

No retries should look like separate completed runs.

---

## 11. PROOF NAMING RULE
Use fixed simple naming.

Preferred pattern:
- _proof/[phase_name]/
- _proof/[phase_name].zip

Inside proof folder, preferred files:
- summary.txt
- validation.txt
- runner_stdout.txt
- runner_stderr.txt

Only include files that are actually needed.
Do not create all four automatically unless the task genuinely needs them.

---

## 12. MAXIMUM PROOF DISCIPLINE
Maximum final deliverables per run:
- 1 proof folder
- 1 proof zip

That is the cap.

If agent believes more are needed:
- STOP first
- explain why
- do not generate them without approval

---

## 13. VALIDATION RULE
A phase is PASS only if the requested validation actually ran and actually passed.

PASS is forbidden when:
- code changed but validation did not run
- validation crashed
- markers are missing
- output is ambiguous
- proof artifacts are incomplete
- wrong root was used
- stale binary was used when current build was required

When in doubt:
- FAIL, not PASS

---

## 14. STALE / WRONG BINARY RULE
Never claim success from stale artifacts.

Before PASS:
- confirm the evidence corresponds to the current requested run
- if isolation/build requirements were specified, honor them exactly

If not proven current:
- FAIL

---

## 15. RUNNER BEHAVIOR RULE
A phase runner must:
- run the intended task only
- emit deterministic results
- produce final validation markers
- return a clean PASS/FAIL outcome
- avoid branching into alternate behaviors

Runner must not:
- silently skip failed steps
- continue after critical failure
- create multiple proof variants
- auto-chain into another phase

---

## 16. MARKER RULE
Markers must be specific to the current phase.
Use YES/NO only where possible.

Bad:
- everything_looks_good=YES

Good:
- root_verified=YES
- build_completed=YES
- viewport_reconcile_correct=YES

Only include markers that matter to the current task.

---

## 17. NO CHATTER RULE
Do not produce long explanations unless asked.

Default output should be:
- direct
- short
- auditable
- phase-focused

No motivational language.
No planning essay.
No repeated context dump.

---

## 18. NO HIDDEN WORK RULE
All actions must be visible in terminal output or final proof artifacts.
No hidden execution panels.
No silent background work.

If an action is not auditable:
- do not do it

---

## 19. RETRY RULE
Do not create multiple “final” attempts.

If a retry is necessary:
- overwrite or quarantine prior temporary attempt
- keep only the final accepted run as final evidence

Do not leave:
- final_v2
- final_v3
- latest_final
- final_fixed
- final_real

One final means one final.

---

## 20. STOP RULE
After reporting the result:
- STOP

Do not continue to:
- next phase
- extra cleanup
- bonus hardening
- additional analysis
unless explicitly instructed.

---

## 21. DEFAULT TASK HEADER
When given a task, interpret it in this structure:

APP: [name]
PHASE: [name/number]
STATE: [current fail/pass state]
OBJECTIVE: [single task]
INPUT: [logs/code]
PROOF LIMIT: 1 folder + 1 zip max

If some fields are missing, only act on what is clearly requested.
Do not invent extra scope.

---

## 22. ENFORCEMENT RULE
If these rules conflict with “being helpful,” these rules win.

More output is not better.
More files is not better.
More changes is not better.

Correct, minimal, auditable, repeatable wins.

---

## 23. HARD VIOLATIONS
Any of these force FAIL:
- wrong root
- hidden execution
- multiple final proof sets
- PASS without real validation
- unrelated file changes
- chaining into additional tasks
- stale evidence presented as current
- more than 1 final proof folder and 1 final zip

---

## 24. SUCCESS DEFINITION
A successful run means:
- correct root
- scoped task only
- minimal code change
- validation actually ran
- markers clearly show outcome
- one final proof folder
- one final proof zip
- clean PASS/FAIL block
- no babysitting required
